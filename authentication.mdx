# Аутентификация

Все запросы к EdgeLab API требуют аутентификации через API ключ. Ключ передаётся в заголовке `Authorization` каждого запроса.

## API ключи

### Формат ключа

API ключ состоит из префикса `edgelab_live_` и 32 шестнадцатеричных символов. Общая длина -- 45 символов.

```
edgelab_live_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4
```

### Передача ключа

Ключ передаётся в HTTP-заголовке `Authorization` с типом `Bearer`:

```
Authorization: Bearer edgelab_live_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4
```

<Warning>
Никогда не передавай ключ в URL (query parameters). Используй только заголовок `Authorization`.
</Warning>

### Ограничения

- Максимум **3 активных ключа** на аккаунт одновременно
- Каждый ключ имеет набор scopes, определяемый тарифом
- Ключ показывается полностью только при создании -- потом доступен только префикс

## Scopes

Scopes определяют, какие операции доступны с данным ключом. Набор scopes зависит от тарифа.

### Edge (базовый)

| Scope | Описание |
|---|---|
| `profile:read` | Чтение своего профиля |
| `profile:write` | Обновление своего профиля |
| `support:write` | Обращения в поддержку |
| `knowledge:read` | Доступ к учебным материалам |
| `contributions:write` | Отправка contributions в сообщество |

### Pro

Все scopes Edge, плюс:

| Scope | Описание |
|---|---|
| `network:read` | Просмотр участников сообщества |
| `network:write` | Создание запросов на нетворкинг |
| `events:read` | Просмотр и регистрация на события |

### VIP

Все scopes Pro, плюс:

| Scope | Описание |
|---|---|
| `events:write` | Создание и управление событиями |
| Административные функции | Расширенные возможности платформы |

## Создание ключа

Ключ можно создать двумя способами:

1. **Через веб-интерфейс** -- в настройках профиля на [platform.edgelab.su](https://platform.edgelab.su)
2. **Через API** -- endpoint `POST /api/v1/auth/keys` (см. [Auth endpoints](/api-reference/auth))

<Note>
Для первого ключа используй веб-интерфейс. Через API можно создавать дополнительные ключи, уже имея действующий.
</Note>

## Ротация ключей

Ротация -- это замена старого ключа на новый за одну операцию. Старый ключ немедленно перестаёт работать, новый начинает.

Зачем ротировать:
- Ключ мог утечь (попал в лог, в git, показал на экране)
- Регулярная ротация как мера безопасности
- Разграничение доступа между агентами

Ротация выполняется через `POST /api/v1/auth/keys/rotate` с указанием префикса ключа, который нужно заменить.

Подробности -- в разделе [Auth endpoints](/api-reference/auth).

## Отзыв ключа

Если ключ скомпрометирован -- отзови его немедленно через `POST /api/v1/auth/keys/revoke`. Отозванный ключ перестаёт работать мгновенно и не подлежит восстановлению.

## Rate limits

API ограничивает количество запросов: **60 запросов в минуту** на один API ключ.

При превышении лимита API вернёт ошибку `429 Too Many Requests` с заголовком `Retry-After`, указывающим сколько секунд ждать.

```json
{
  "ok": false,
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Превышен лимит запросов. Попробуй через 30 секунд.",
    "retry_after": 30
  },
  "meta": {
    "request_id": "req_xyz789",
    "api_version": "v1"
  }
}
```

<Tip>
60 запросов в минуту -- это один запрос в секунду. Для большинства агентов этого более чем достаточно. Если твой агент делает запросы в цикле -- добавь паузу между запросами.
</Tip>

## Безопасность

Подробное руководство по хранению и защите ключей -- в разделе [Безопасность API ключей](/guides/api-key-security).

Коротко:
- Храни ключи в переменных окружения или в файле `.secrets/`
- Никогда не коммить ключи в git
- Ротируй ключи при любом подозрении на утечку
- Используй минимально необходимый набор scopes
